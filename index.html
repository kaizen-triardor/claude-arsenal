Memory System v2 — 4-Tier Architecture

A unified persistent-memory stack across Obsidian, MemPalace, and Cognee. Designed 2026-05-17, Tier 4 completed 2026-05-18.

TL;DR

4 cooperating stores, each with one job. Tier 1 is the always-loaded index (~95 lines, ~2K tokens). Tier 2 is a curated Obsidian vault of decisions, sessions, and references. Tier 3 is MemPalace — every Claude Code conversation mined automatically into a verbatim store. Tier 4 is Cognee — three datasets giving chunk recall, slug-keyed graph traversal, and natural-language graph queries via an LLM. The join key across all tiers is the entity slug — same name = same entity everywhere.

The Four Tiers

TIER 1

HOT — Always loaded

MEMORY.md in Claude Code's per-project memory directory

  • ~95 lines, ~2K tokens, injected into every session by Claude Code
  • Index only: one line per topic + pointer to the detail file
  • Sections: active items · paused · archived · workflow rules · references · sessions · meta

Job: tell Claude what exists. Detail lives elsewhere.

TIER 2

WARM — Curated vault

Obsidian vault, hundreds of markdown files. Vault lives on the OS-native filesystem (symlinked when Linux/WSL talks to it from a Windows-side Obsidian.app).

  • 10-Projects-Active/ — one file per live item
  • 20-Projects-Paused/ · 30-Projects-Archived/
  • 40-Sessions/ — dated session logs
  • 50-Feedback/ — workflow rules + decisions
  • 60-Reference/ — configs, install notes, credential pointers
  • 70-People/ · 90-Meta/ · 00-Inbox/

Job: rich, human-readable narrative. Read on demand via Read.

TIER 3

EPISODIC — MemPalace

~/.mempalace/ · ChromaDB + SQLite knowledge graph · tens of thousands of drawers

  • Every Claude Code session JSONL transcript auto-mined
  • Wings = working directory; Rooms = extracted memory type (decision, problem, milestone, preference)
  • Hybrid scheduling: SessionEnd hook + 6h systemd timer + hourly health watchdog
  • flock-protected · size:mtime idempotency · alert dropped to vault inbox if stale >48h

Job: "what did we discuss about X?" Answers from verbatim history.

TIER 4

SEMANTIC — Cognee · 3 datasets

~/.cognee/ · LanceDB + Ladybug + SQLite

  • Chunks dataset — embeddings of the whole vault via local nomic-embed-text. CHUNKS search.
  • Subagent graph dataset — entities + edges built one-shot by ~10 parallel Claude Code subagents with a controlled vocabulary. $0.
  • Cognified graph dataset — native cognify() via a cloud LLM on a free tier. Powers GRAPH_COMPLETION natural-language queries. Daily refresh via systemd timer.

Job: cross-cutting questions like "which items share infrastructure X?" or "what's blocking item Y?" Answered in 1-2 seconds.

How a Query Flows

User asks: "What credentials exist for <item>?"

  1. Tier 1 (free) — MEMORY.md is already in context. Index shows
     the item exists and where its detail file lives.
     Cost: 0 extra tokens.

  2. Tier 2 → Read 60-Reference/<the credential pointer>.md
     when the question is targeted. ~2K tokens for the specific file.

  3. Tier 4 → cognee.search(q, SearchType.GRAPH_COMPLETION)
     when the question spans items. The LLM traverses the graph and
     returns a natural-language answer with cited entities. ~500 in,
     ~200 out, 1-2s latency.

  4. Tier 3 → mempalace_search(q) when the question is
     historical ("what did we decide about X in March?").
     Returns matching conversation chunks with timestamps.

The 4 tiers compose. Same slug resolves the same entity across all
four stores — that's the join key.

Pros & Cons

PROS

  • Persistent across sessions. New Claude session = full context loaded in 2K tokens.
  • No re-explaining. "What is <item>?" answers survive across days and machines.
  • Cross-cutting recall. "Which items share infrastructure X?" answers in 1 query.
  • Autonomous maintenance. MemPalace mines on every session end; the graph refreshes daily.
  • $0 cost. Local embeddings + free-tier cloud LLM + Claude Max plan for subagent extraction.
  • No corruption path. Cloud LLM means no graph-DB migration bugs; subagents bypass cognify() entirely.
  • Idempotent + resumable. Daily timer picks up where yesterday's quota ran out.
  • Obsidian-native. Vault is browseable, editable, searchable in Obsidian.app.

CONS

  • Setup is non-trivial. 4 stores, ~6 scripts, ~3 systemd units, 2 Claude Code hooks. Worth it once running.
  • Stale data risk. A memory written weeks ago about code may no longer reflect reality. Mitigation: verify against current code before asserting.
  • Naming drift. If an item is renamed but the vault slug isn't, the graph join breaks. Manual cleanup needed.
  • Free-tier daily ceiling. Cloud LLM limits cap how fast big changes propagate — a full refresh may take 2-3 days after a major vault rewrite.
  • Tier 4 cognify isn't bit-for-bit idempotent. Re-running on the same files produces slightly different entities (LLM variance). State file mitigates the rerun cost.
  • Single-machine. Portable to other machines only with the symlink and systemd units redeployed.
  • Memory ≠ truth. Always verify before acting — a memory is a point-in-time observation, not live state.

Failure Modes & Recovery

Symptom: MemPalace health alert in vault inbox

→ Mining stalled >48h. Inspect the mining service logs, then manually run the full-sweep script.

Symptom: Cognee refresh alert in vault inbox

→ Refresh failed >72h. Check the refresh log; if it's a quota-only stop, wait for the next UTC day reset.

Symptom: cognee.recall returns "NoneType.id"

→ MCP server started before any user existed. Restart Claude Code session.

Symptom: a memory contradicts reality

→ Memory ≠ truth. Verify against current code, then update or delete the stale memory.
